## Pre-Deployment Checklist

### Security
- [ ] Change `JWT_SECRET` to a strong random value (32+ characters)
- [ ] Change `ADMIN_SECRET_KEY` to a strong random value
- [ ] Set `DEBUG=false`
- [ ] Set `ENVIRONMENT=production`
- [ ] Configure `ALLOWED_ORIGINS` with your production domains only
- [ ] Verify API keys are set (ANTHROPIC_API_KEY, etc.)
- [ ] Ensure HTTPS is configured

### Database
- [ ] PostgreSQL is set up (not SQLite for production)
- [ ] `AUTO_CREATE_TABLES=false`
- [ ] All migrations are tested on staging
- [ ] Backup system is configured and tested

### Infrastructure
- [ ] Server resources are adequate (CPU, RAM, Storage)
- [ ] Monitoring is set up (logs, metrics)
- [ ] SSL certificate is installed
- [ ] Firewall rules are configured

## Environment Configuration

### Production `.env` File

```bash
# Application
APP_NAME=Gen-Friend
DEBUG=false
ENVIRONMENT=production
LOG_LEVEL=INFO

# Database (PostgreSQL for production)
DATABASE_URL=postgresql+asyncpg://user:password@host:5432/genfriend
AUTO_CREATE_TABLES=false

# Redis (optional but recommended)
REDIS_URL=redis://localhost:6379
REDIS_ENABLED=true

# Security
JWT_SECRET=your-very-long-random-secret-key-here-at-least-32-chars
JWT_ALGORITHM=HS256
ACCESS_TOKEN_EXPIRE_MINUTES=1440

# Admin
ADMIN_SECRET_KEY=another-very-secure-random-key

# AI Services
ANTHROPIC_API_KEY=sk-ant-xxx
OPENAI_API_KEY=sk-xxx
GROQ_API_KEY=gsk_xxx

# Provider Selection
STT_PROVIDER=groq
LLM_PROVIDER=anthropic
TTS_PROVIDER=openai

# CORS (comma-separated production domains)
ALLOWED_ORIGINS=https://yourdomain.com,https://www.yourdomain.com

# Authentication (if using Clerk)
CLERK_SECRET_KEY=sk_xxx
CLERK_PUBLISHABLE_KEY=pk_xxx

# Backup
BACKUP_ENABLED=true
BACKUP_PATH=/var/backups/genfriend
BACKUP_RETENTION_DAYS=30
```

### Key Differences from Development

| Setting | Development | Production |
|---------|-------------|------------|
| `DEBUG` | true | false |
| `ENVIRONMENT` | development | production |
| `AUTO_CREATE_TABLES` | true | **false** |
| `DATABASE_URL` | SQLite | PostgreSQL |
| `ALLOWED_ORIGINS` | localhost:3002 | Your domains |
| `JWT_SECRET` | dev-secret | Strong random key |

## Database Strategy

### Golden Rule
> **Never use `create_all()` or drop tables in production. Always use Alembic migrations.**

### How It Works

**Development:**
- `AUTO_CREATE_TABLES=true` (default)
- Tables created automatically on startup
- Quick iteration, no migration overhead

**Production:**
- `ENVIRONMENT=production` or `AUTO_CREATE_TABLES=false`
- Tables **never** created automatically
- All schema changes through Alembic migrations

### Migration Workflow

#### 1. Generate Migration (Development)
```bash
cd backend

# After modifying models, generate migration
alembic revision --autogenerate -m "describe_the_change"

# Review the generated file in alembic/versions/
# IMPORTANT: Always review before applying!
```

#### 2. Test Migration (Staging)
```bash
# Apply to staging database
alembic upgrade head

# Verify everything works
# Run tests
```

#### 3. Deploy to Production
```bash
# 1. Create backup FIRST
python scripts/backup.py backup

# 2. Apply migration
alembic upgrade head

# 3. Verify
alembic current
```

### Common Migration Commands

```bash
# Show current migration
alembic current

# Show migration history
alembic history

# Apply all pending migrations
alembic upgrade head

# Apply specific migration
alembic upgrade <revision_id>

# Rollback one migration
alembic downgrade -1

# Rollback to specific revision
alembic downgrade <revision_id>

# Show pending migrations (not yet applied)
alembic upgrade head --sql
```

### Migration Best Practices

1. **One change per migration** - Easier to rollback
2. **Test rollback** - Ensure `downgrade()` works
3. **Never edit applied migrations** - Create new ones
4. **Backup before migration** - Always!
5. **Review autogenerated migrations** - Alembic can miss things

## Backup Strategy

### Backup Types

| Type | Frequency | Retention | Use Case |
|------|-----------|-----------|----------|
| **Full Backup** | Daily | 30 days | Complete recovery |
| **Pre-Deployment** | Before each deploy | 7 days | Quick rollback |
| **On-Demand** | As needed | Keep indefinitely | Major changes |

### Using the Backup Script

```bash
cd backend

# Create backup
python scripts/backup.py backup
# Output: Backup created: ./backups/genfriend_backup_20240115_143022.sql

# List backups
python scripts/backup.py list

# Restore from backup (requires confirmation)
python scripts/backup.py restore ./backups/genfriend_backup_20240115_143022.sql

# Clean up old backups
python scripts/backup.py cleanup
```

### Automated Daily Backups

#### Option 1: Cron Job (Linux)
```bash
# Edit crontab
crontab -e

# Add daily backup at 2 AM
0 2 * * * cd /path/to/backend && /path/to/python scripts/backup.py backup >> /var/log/genfriend-backup.log 2>&1

# Add weekly cleanup (Sundays at 3 AM)
0 3 * * 0 cd /path/to/backend && /path/to/python scripts/backup.py cleanup >> /var/log/genfriend-backup.log 2>&1
```

#### Option 2: Systemd Timer (Linux)
```bash
# /etc/systemd/system/genfriend-backup.service
[Unit]
Description=Gen-Friend Database Backup

[Service]
Type=oneshot
WorkingDirectory=/path/to/backend
ExecStart=/path/to/python scripts/backup.py backup
User=genfriend

# /etc/systemd/system/genfriend-backup.timer
[Unit]
Description=Daily Gen-Friend Backup

[Timer]
OnCalendar=*-*-* 02:00:00
Persistent=true

[Install]
WantedBy=timers.target
```

Enable:
```bash
sudo systemctl enable genfriend-backup.timer
sudo systemctl start genfriend-backup.timer
```

#### Option 3: Cloud Provider Backups

**AWS RDS:**
- Automated backups enabled by default
- Configure retention period (1-35 days)
- Point-in-time recovery available

**DigitalOcean Managed Database:**
- Daily automatic backups
- 7-day retention (configurable)
- One-click restore

**Render/Railway:**
- Automatic daily backups
- Manual backup available via dashboard

### Backup Storage Recommendations

| Environment | Storage Location |
|-------------|------------------|
| Local/VPS | Separate volume or remote storage |
| AWS | S3 bucket with lifecycle policy |
| GCP | Cloud Storage bucket |
| Azure | Blob Storage |

### Backup Verification

Monthly: Restore backup to a test database and verify data integrity.

```bash
# Create test database
createdb genfriend_test

# Restore backup
pg_restore -d genfriend_test backup_file.sql

# Connect and verify
psql genfriend_test -c "SELECT COUNT(*) FROM users;"
psql genfriend_test -c "SELECT COUNT(*) FROM goals;"

# Cleanup
dropdb genfriend_test
```

## Deployment Steps

### First-Time Deployment

```bash
# 1. Set up server and PostgreSQL

# 2. Clone repository
git clone <repo> /opt/genfriend
cd /opt/genfriend/backend

# 3. Create virtual environment
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 4. Configure environment
cp .env.example .env
# Edit .env with production values

# 5. Create database
createdb genfriend

# 6. Run migrations (creates all tables)
alembic upgrade head

# 7. Start application
uvicorn app.main:app --host 0.0.0.0 --port 8001
```

### Subsequent Deployments

```bash
# 1. Create backup
python scripts/backup.py backup

# 2. Pull latest code
git pull origin main

# 3. Update dependencies (if changed)
pip install -r requirements.txt

# 4. Run migrations (if any)
alembic upgrade head

# 5. Restart application
sudo systemctl restart genfriend
```

### Zero-Downtime Deployment

For zero-downtime, use a process manager like:

**Using Gunicorn + Supervisor:**
```bash
# Graceful reload
sudo supervisorctl signal HUP genfriend
```

**Using systemd:**
```bash
# Graceful restart
sudo systemctl reload genfriend
```

## Rollback Procedures

### Code Rollback

```bash
# 1. Identify previous working commit
git log --oneline

# 2. Revert to previous version
git checkout <previous_commit>

# 3. Restart application
sudo systemctl restart genfriend
```

### Database Rollback (Migration)

```bash
# 1. Identify current migration
alembic current

# 2. Rollback one migration
alembic downgrade -1

# 3. Or rollback to specific version
alembic downgrade <revision_id>
```

### Full Database Restore

```bash
# 1. Stop application
sudo systemctl stop genfriend

# 2. Restore from backup
python scripts/backup.py restore ./backups/genfriend_backup_YYYYMMDD_HHMMSS.sql

# 3. Verify migration state
alembic current

# 4. Start application
sudo systemctl start genfriend
```

### Emergency Procedures

**If production is down:**

1. Check logs: `tail -f /var/log/genfriend/app.log`
2. Check database: `psql genfriend -c "SELECT 1;"`
3. Check disk space: `df -h`
4. Check memory: `free -m`
5. Rollback to last known good state

**If data corruption suspected:**

1. Stop application immediately
2. Create backup of current state (even if corrupted)
3. Restore from last known good backup
4. Investigate cause before resuming
